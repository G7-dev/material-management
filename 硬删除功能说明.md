# 硬删除功能说明

## 重要更新

从软删除改为硬删除，数据将直接从数据库中永久删除，无法恢复！

## 更改说明

### 什么是硬删除？

硬删除是指直接从数据库中删除记录，而不是将状态标记为'inactive'。删除后数据将无法恢复。

### 更改内容

#### 1. 删除物资（硬删除）

**修改文件**：`src/app/utils/materialsDB.ts`

**函数**：`deleteMaterial()`

**更改前**：
```typescript
const { error } = await supabase
  .from('materials')
  .update({ status: 'inactive' })
  .eq('id', materialId);
```

**更改后**：
```typescript
const { error } = await supabase
  .from('materials')
  .delete()
  .eq('id', materialId);
```

**影响**：
- 物品将从 `materials` 表中永久删除
- 无法恢复被删除的数据
- 相关的库存流水记录仍然保留（用于审计）

#### 2. 删除规格（硬删除）

**修改文件**：`src/app/utils/materialsDB.ts`

**函数**：`deleteMaterialSize()` 和 `deleteMaterialSizes()`

**说明**：
- 这些函数从 `sizes` JSON 数组中移除规格
- 通过 `jsonb_set` 和 `filter` 操作更新数据库
- 规格数据从 JSON 数组中永久移除

#### 3. UI 文本更新

**修改文件**：
- `src/app/pages/ItemPermission.tsx`
- `src/app/pages/LowStockAlert.tsx`

**更改内容**：
- 将"软删除"改为"永久删除"
- 添加警告提示："数据将从数据库中彻底删除，无法恢复"
- 更新确认对话框的描述

**警告文本示例**：
```
⚠️ 警告：此操作将从数据库中永久删除该物品及其所有规格，无法恢复！
```

#### 4. 移除不必要的过滤

**修改文件**：`src/app/utils/materialsDB.ts`

**函数**：`fetchMaterials()`

**更改前**：
```typescript
const { data, error } = await supabase
  .from('materials')
  .select('*')
  .eq('status', 'active')  // 只获取有效的物品
  .order('updated_at', { ascending: false });
```

**更改后**：
```typescript
const { data, error } = await supabase
  .from('materials')
  .select('*')
  .order('updated_at', { ascending: false });
```

**原因**：硬删除后，已删除的物品不再存在于数据库中，无需过滤。

## 功能影响

### 删除整个物品

**操作路径**：
1. 打开物品权限管理页面
2. 点击物品的垃圾桶图标
3. 点击"删除整个物品"

**结果**：
- ✅ 物品从 `materials` 表中永久删除
- ✅ 物品的规格数据被删除
- ✅ 物品从列表中消失
- ✅ 刷新页面后仍然不会出现
- ❌ 无法恢复已删除的物品

### 删除规格

**操作路径**：
1. 打开物品权限管理页面
2. 点击物品的垃圾桶图标
3. 选择要删除的规格
4. 点击"删除选中规格"

**结果**：
- ✅ 选中的规格从 `sizes` JSON 数组中移除
- ✅ 总库存重新计算
- ✅ 规格从界面中消失
- ❌ 无法恢复已删除的规格

### 编辑模态框删除

**操作路径**：
1. 打开低库存预警页面
2. 点击物品的"编辑"按钮
3. 点击"删除整个物品"

**结果**：
- ✅ 物品从数据库中永久删除
- ✅ 编辑模态框关闭
- ✅ 列表自动刷新

## 数据保留策略

### 保留的数据

以下数据在删除物品后会保留：

1. **库存流水记录** (`inventory_logs`)
   - 保留所有历史操作记录
   - 用于审计和追踪
   - `material_id` 字段保留，但对应的物品已不存在

2. **申请记录** (`requisitions`)
   - 保留所有申请历史
   - 用于报表和审计

### 删除的数据

以下数据会被永久删除：

1. **物品基本信息**
   - 名称、分类、规格等
   
2. **库存信息**
   - 当前库存、预警阈值
   - 规格信息（如果是删除整个物品）

3. **图片信息**
   - 物品图片 URL（如果存储在数据库中）

## 注意事项

### ⚠️ 重要警告

1. **数据无法恢复**
   - 删除操作是永久的
   - 没有回收站功能
   - 删除前请仔细确认

2. **审计日志保留**
   - 虽然物品被删除，但操作记录保留
   - 可以通过日志追踪删除操作

3. **外键关联**
   - 删除物品后，`inventory_logs` 中的 `material_id` 将成为孤立引用
   - 查询时需要注意处理不存在的情况

### 💡 建议

1. **备份数据**
   - 定期备份数据库
   - 重要删除前手动备份

2. **权限控制**
   - 限制删除权限，只给管理员
   - 添加操作日志

3. **二次确认**
   - 删除操作需要二次确认
   - 显示详细的删除影响范围

4. **测试环境**
   - 在生产环境删除前，先在测试环境验证

## 性能影响

### 正面影响

✅ **查询性能提升**
- 无需过滤 `status` 字段
- 表数据量减少（已删除的数据不再占用空间）

✅ **存储空间节省**
- 已删除的数据不再占用存储空间
- 减少数据库备份大小

### 负面影响

❌ **数据丢失风险**
- 误删除无法恢复
- 需要更谨慎的操作

❌ **审计复杂性**
- 需要保留更多日志来追踪删除操作
- 跨表查询时需要处理孤立引用

## 安全建议

1. **操作日志**
   - 记录所有删除操作
   - 包括：操作人、时间、删除的物品、原因

2. **权限审查**
   - 定期审查删除权限
   - 限制只有必要的人员可以删除

3. **数据备份**
   - 每日自动备份
   - 重要操作前手动备份

4. **监控告警**
   - 监控大量删除操作
   - 异常删除发送告警

## 总结

硬删除功能提供了更彻底的数据清理能力，但也带来了数据不可恢复的风险。请确保：

- ✅ 理解硬删除的后果
- ✅ 有足够的备份策略
- ✅ 有严格的权限控制
- ✅ 有完整的操作日志

如有疑问，建议先咨询数据库管理员或技术负责人。
